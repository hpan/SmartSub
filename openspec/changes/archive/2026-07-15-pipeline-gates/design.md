# Design: 人工闸门（P3）

## Context

- P2 的附加阶段模型：`processFile` 尾部顺序执行 dub/compose；阶段状态字符串字段 + 事件镜像持久化；带附加阶段任务的重试可复用上游产物续跑（resume 判定）；任务配置随 `workItem.configSnapshot` 快照执行。
- 校对编辑器（任务页内嵌 ProofreadEditor）保存时经 sidecar 重写交付物；P2 的配音文本源以字幕内容 hash 判定会话可恢复性——**校对修改字幕 → hash 变化 → 配音会话自动重建**，闸门语义天然衔接。
- 配音工作台已支持按 sessionId 恢复会话（P1），流水线文件的 `dubbingSessionId` 已落文件字段（P2）。
- 探索定稿：文件级流式（到闸即可校对）、界面白话不出现"闸门"、含配音/合成默认开字幕校对、配音确认默认自动。

## Goals / Non-Goals

**Goals:**

- 两个人工检查点（字幕校对/配音确认），各可 `manual|auto`，文件级独立停靠与放行。
- 停靠不占资源、跨重启保留；放行续跑不重做已完成工作；校对修改自动向下游传导（重配/重建音轨）。
- 检查工具复用既有工作台（内嵌校对编辑器 / 配音工作台），以"检查员包壳"叠加放行动线。
- 未配置 gates 的任务零行为变化。

**Non-Goals:**

- 不新增第三类闸门（如合成前预览）；不做批注/驳回工作流（放行是唯一通过动作，修改即校对工具内完成）。
- 不做配方/启动台改版（P4）。

## Decisions

### D1. 停靠模型：到闸即结束本轮，放行即重新派发（不引入等待调度）

到闸的文件把闸门字段置 `review` 并**正常结束本轮 `processFile`**（非 error 非 done）；放行 = 置 `passed` + 按快照重新派发该文件，P2 的续跑判定让已完成段落全部跳过、从闸门后继续。

- **为什么不用"在 processFile 内 await 放行"的等待模型**：等待占用并发槽会堵住其它文件；释放槽再等待需要新的调度状态机；且进程重启后等待丢失。停靠模型复用"重试即续跑"的全部现有机制——持久化、跨重启、取消、进度展示零新概念。
- 闸门字段：`IFiles.subtitleGate` / `dubbingGate`，取值 `'' | 'review' | 'passed'`。`review` 不参与启动时的中断标记（`STAGE_KEYS` 不含闸门字段），跨重启天然保留。
- 检查点位置：字幕校对在字幕段（含翻译）成功之后、配音/合成之前；配音确认在配音阶段成功之后、合成之前。仅当任务存在对应下游阶段时闸门才有意义（向导只在勾了配音或成品视频时显示把关开关）。

### D2. 放行经主进程统一入口（工作台/任务页同一 IPC）

新增 `pipeline:releaseGate { projectId, gate, fileUuids? }`：主进程取 workItem → 过滤处于 `review` 的目标文件 → 置 `passed` 并落库 → 以 `configSnapshot` 为 formData 重新派发这些文件。

- 派发复用 `taskProcessor` 队列：导出主进程内部入口 `enqueueProjectFiles(projectId, files, formData)`，用已持有的 `progressWindow.webContents` 合成事件 sender（`wrapTaskEvent` 兼容），不依赖渲染层再拼 `handleTask`。
- 配音工作台的「放行并继续」与任务页的「放行/全部放行」都走该 IPC，语义单点。

### D3. 校对修改与配音修改的下游传导

- **字幕校对后放行**：校对保存已重写交付物与 sidecar → 配音文本源（纯译文/ sidecar）内容变化 → `materializeDubSubtitle` 产出新内容 → 会话 hash 不匹配 → dubStage 自动重建会话重配。文本未改则会话恢复、已合成行原样复用。零新代码，P2 机制的自然结果。
- **配音确认中重生成行后放行**：会话行级状态已持久化；`resume.dubbingDone` 的跳过逻辑调整为「跳过批量合成但**总是重建配音轨**」（`buildDubTrack` 为纯拼接，秒级），保证工作台里的修改进入成片。原实现直接复用旧 track，属 P3 必须修正的点。

### D4. 到闸通知与完成通知联动

- 新增 `gateNotifier`：文件到闸时按「任务×闸门」聚合节流（5s），系统通知「《任务名》有 N 个文件等待字幕校对/配音确认」，点击聚焦窗口。
- `finalizeProjectIfDrained` 的完成通知：工程排空时若存在 `review` 文件，通知文案改为「有文件等待校对」而非「任务全部完成」；任务列表状态显示为等待校对（workItem 状态推导把 review 视为非完成、非错误的进行中间态——`derivePipelineWorkItemStatus` 增加 review 感知，展示层映射为"待校对"）。

### D5. 检查员 UX：包壳复用，不改编辑器内核

- **任务详情**：轨道在阶段之间渲染菱形检查点（pending 灰 / review 高亮 / passed 打勾）；顶部聚合操作条按闸门统计 review 文件数，动作 = 逐个校对（打开首个待校文件的检查工具）/ 全部放行。字幕检查点行级动作：「校对」（内嵌 ProofreadEditor，外壳底部叠加「放行并继续下一个」——保存由编辑器自身负责，放行动作关闭编辑器、调用 releaseGate、自动打开下一个待校文件）与「放行」。
- **配音检查点**：行级/聚合「检查配音」→ 跳配音工作台 `?session=<id>&gateProject=<pid>&gateFile=<uuid>`；工作台顶部检查员条：任务名与待检序号、上一个/下一个（切换同任务其它 review 文件的会话）、「放行并继续」（releaseGate 单文件 → 有下一个则切换，否则返回任务页）、「全部放行」。工作台既有的批量/单行操作全部可用；检查员模式下隐藏「导出」（产出由合成阶段负责）。

### D6. 向导「人工把关」开关与默认值

勾选配音或成品视频时显示「人工把关」区：`字幕校对`（默认**开**）与 `配音确认`（仅勾了配音时显示，默认关）。白话文案 + 一句代价说明（"到点会提醒你，校对后继续"）。`gates` 随 formData 入快照。

## Risks / Trade-offs

- [用户忽略通知，任务"停着不动"被误解为卡死] → 任务列表待校对状态高亮 + 聚合操作条常驻 + 启动台工作项状态显示"待校对"；通知点击聚焦。
- [全部放行误触浪费云端配音] → 全部放行二次确认（含文件数）；字幕闸门默认开本身就是成本护栏。
- [releaseGate 与用户手动"开始/重试"并发重派发同一文件] → 派发前校验闸门状态（仅 review→passed 才派发）；processFile 对 gate=passed 幂等（重复跑续跑跳过）。
- [配音检查点导航跨页状态（上/下一个）依赖任务快照实时性] → 检查员条每次动作前重新拉取 workItem 计算待检清单，不缓存。
- [review 状态文件参与「重试失败」误放行] → 重试入口只针对 error 文件；review 文件的唯一动作是校对/放行。

## Migration Plan

1. 类型与字段（零行为变化）。
2. 主进程停靠+放行+通知（gates 未配置时代码路径不激活）。
3. 任务详情 UX → 配音工作台检查员模式 → 向导开关 + i18n。
4. 回滚：各步独立；gates 缺省 auto 等价于 P2 现状。

## Open Questions

- 全部放行的确认强度（对话框 vs 二次点击）——实现期按对话框先做。
- 配音确认默认值是否随用户使用数据调整（P4 配方可按配方粒度记忆）。
